# PTAC-NO-GENERATED-BLOBS — No Generated Blobs

**Layer:** 2
**Categories**: version-control, reproducibility, transparency
**Applies-to**: all
**Summary:** Commit source and generation instructions, not generated or binary output blobs.

## Principle

Version control tracks *intent*, not *output*. Commit the source files and the instructions to generate them; let the build system produce the output. Generated files that are committed must be clearly marked, kept in sync, and justified by a concrete consumer need.

## Why it matters

Generated files committed without discipline cause constant noise: every regeneration produces a diff with no semantic content. Binary outputs inflate repository size, cannot be reviewed meaningfully, and become stale. When source and generated output diverge, the repo tells a lie — the committed output is not what the current source would produce. AI-generated content committed without review falls into the same trap.

## Violations to detect

- Compiled binaries (`.jar`, `.exe`, `.wasm`) committed to the repo
- Generated API clients or SDKs checked in without a `.gitattributes` or `# generated` marker
- Lock files for generated assets (e.g., compiled CSS from Sass) without a `.gitignore` entry for the source's output dir
- `package-lock.json` or `yarn.lock` committed in libraries (not applications) without justification
- AI-generated code or documentation committed wholesale without review markers

## Inspection

- `find $TARGET -name "*.jar" -o -name "*.exe" -o -name "*.wasm" | head -20` | MEDIUM | Binary executables committed to repo

## Good practice

```
# Commit this:
src/api/openapi.yaml        ← source
scripts/generate-client.sh  ← generation instructions

# Not this:
src/generated/ApiClient.java  ← generated output (commit only if required by consumer)

# If you must commit generated output, mark it:
# Generated by scripts/generate-client.sh — do not edit manually
```
